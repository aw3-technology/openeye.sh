## 181. Model Registry

**As an enterprise ML team, I can use a centralized model registry for versioning, storing, and serving detection models with metadata tracking.**

### Acceptance Criteria

- [ ] `openeye model register --name warehouse-safety --version 1.0.0 --file model.onnx --format onnx` registers a model in the centralized registry
- [ ] `openeye model list` displays all registered models with name, version, format, size, creation date, author, and status (active/deprecated/archived)
- [ ] `openeye model info warehouse-safety:1.0.0` shows full metadata: architecture, input/output shapes, training dataset, mAP, parameters, FLOPs, license, tags, and custom key-value annotations
- [ ] `openeye model pull warehouse-safety:latest` downloads the latest version of a model from the registry to the local cache at `~/.openeye/models/`
- [ ] `openeye serve --model registry://warehouse-safety:1.0.0` loads a model directly from the registry without a separate pull step
- [ ] Registry supports multiple model formats: ONNX, TensorRT, CoreML, PyTorch (`.pt`), SafeTensors, TFLite, and OpenVINO IR
- [ ] Model metadata is stored as a `model.yaml` manifest alongside the binary artifact, including SHA-256 checksum for integrity verification
- [ ] Registry supports namespaces: `openeye model register --namespace acme-corp --name ...` isolates models per organization
- [ ] Access control: models can be `public`, `org-private`, or `team-private` via `--visibility` flag
- [ ] `openeye model tag warehouse-safety:1.0.0 production` applies mutable tags (e.g., `production`, `staging`, `canary`) for deployment pinning
- [ ] `openeye model delete warehouse-safety:0.9.0` soft-deletes a version (recoverable within 30 days); `--hard-delete` permanently removes it
- [ ] Registry API is exposed at `POST /api/v1/models`, `GET /api/v1/models`, `GET /api/v1/models/{name}/versions/{version}` with full OpenAPI documentation
- [ ] `openeye model search --tag safety --format onnx --min-map 0.75` queries the registry with filters
- [ ] Registry storage backends are configurable: local filesystem, S3, GCS, Azure Blob Storage via `--storage-backend` in server config

### Edge Cases

- [ ] Registering a model with an existing `name:version` combination returns `409 Conflict` with a message: "Version 1.0.0 already exists for warehouse-safety. Use --force to overwrite or increment the version"
- [ ] `--force` overwrite of an existing version creates an audit log entry recording who overwrote it, when, and the previous artifact's checksum — the old artifact is retained for 7 days in a `_overwritten/` prefix
- [ ] Uploading a model >5GB: the CLI uses multipart/chunked upload with resumable uploads. If the connection drops mid-upload, `openeye model register --resume` picks up from the last completed chunk
- [ ] Model checksum mismatch on pull: if the downloaded artifact's SHA-256 doesn't match the manifest, the download is discarded and retried once — if it fails again, exits with a corruption warning
- [ ] Registry storage backend is unreachable (e.g., S3 outage): `openeye model list` falls back to a locally cached index if available (max staleness configurable via `--cache-ttl`, default: 1 hour). `openeye model pull` fails immediately with a clear error — does not serve stale models
- [ ] Concurrent registration of the same `name:version` from two CLI instances: the registry uses optimistic locking (ETags) — the second request receives `409 Conflict`
- [ ] Model with unsupported format: if the file extension does not match any known format and `--format` is not specified, the CLI rejects with a list of supported formats rather than uploading an untyped blob
- [ ] Soft-deleted model pulled by a pipeline: returns `410 Gone` with deletion timestamp and recovery instructions (`openeye model restore warehouse-safety:0.9.0`)
- [ ] Registry index grows beyond 10,000 models: list and search endpoints use cursor-based pagination (`--page-size 50 --cursor <token>`) — no full-table scans
- [ ] Model registered without required metadata fields (e.g., missing `--format`): the CLI prompts interactively or fails with a clear error listing required fields in non-interactive mode
- [ ] Model registered with malicious payload: `.pt` pickle files can contain arbitrary code executed on `torch.load` — no malware/supply-chain scanning on uploaded artifacts
- [ ] Architecture mismatch (ARM vs x86): no metadata enforcement when model compiled for x86 TensorRT is pulled to ARM Jetson — registry stores format but not target architecture
- [ ] Race condition on mutable tag reassignment: two operators simultaneously reassigning `production` tag to different versions — tag operation not covered by optimistic locking

### Technical Notes

- Registry server component lives in `backend/registry/` and exposes a FastAPI-based REST API
- Model artifacts are stored in object storage; metadata is stored in PostgreSQL (or SQLite for single-node deployments)
- The `model.yaml` manifest schema is defined in `backend/registry/schema.py` as a Pydantic model
- CLI commands live in `cli/openeye_ai/commands/model.py`
- The registry is inspired by OCI container registries — models are addressed as `registry://namespace/name:version`
- Storage adapter interface: `RegistryStorage` with `upload(path, artifact)`, `download(path) -> bytes`, `delete(path)`, `exists(path) -> bool`

## 182. Model Versioning & Rollback

**As an enterprise ML team, I can version models with semantic versioning and instantly rollback to previous versions in production.**

### Acceptance Criteria

- [ ] Models in the registry (story 181) follow semantic versioning: `major.minor.patch` (e.g., `2.1.3`) enforced at registration time
- [ ] `openeye model register --name warehouse-safety --version 2.0.0 --breaking-changes "Changed input resolution from 640 to 1280"` documents breaking changes for major version bumps
- [ ] `openeye model versions warehouse-safety` lists all versions with creation date, status, deployment history, and changelog
- [ ] `openeye model promote warehouse-safety:2.1.0 --stage production` transitions a model through stages: `dev` → `staging` → `production`
- [ ] `openeye model rollback warehouse-safety --to 2.0.3` instantly reverts the `production` tag to the specified version and triggers a rolling restart of all serving instances
- [ ] Rollback completes within 30 seconds for models cached on serving nodes; under 2 minutes for models requiring a pull from the registry
- [ ] `openeye model rollback warehouse-safety --to previous` is a shorthand to revert to the version that was `production` before the current one
- [ ] All promotion and rollback events are recorded in an immutable audit log accessible via `openeye model audit warehouse-safety`
- [ ] `openeye model diff warehouse-safety:2.0.3 warehouse-safety:2.1.0` shows metadata differences: mAP delta, parameter count change, training dataset differences, and architecture changes
- [ ] Automatic rollback trigger: `openeye serve --auto-rollback --error-rate-threshold 0.05 --window 5m` reverts to the previous model version if inference error rate exceeds 5% over a 5-minute window
- [ ] Version pinning in deployment configs: `openeye serve --model registry://warehouse-safety:~2.1` resolves to the latest `2.1.x` patch version (semver range syntax)
- [ ] `openeye model deprecate warehouse-safety:1.x` marks all 1.x versions as deprecated — `openeye model pull` on a deprecated version logs a warning with the suggested upgrade path
- [ ] Pre-rollback health check: before applying a rollback, the target version is loaded in a canary instance and validated with a synthetic inference request

### Edge Cases

- [ ] Rollback to a version whose artifact has been hard-deleted: fails with `404 Not Found` and a message explaining that the version was permanently removed — suggests the nearest available version
- [ ] Rollback during an in-flight promotion: the promotion is cancelled, the rollback takes priority, and the audit log records both the cancelled promotion and the rollback with reasons
- [ ] Automatic rollback loop: if the rollback target also produces errors above the threshold, the system halts after one rollback attempt, enters a `DEGRADED` state, and alerts the ops team rather than oscillating between versions
- [ ] Concurrent rollbacks from two operators: the first rollback wins (optimistic lock); the second receives `409 Conflict` with the current state
- [ ] Rollback on a multi-node deployment (story 146): rolling restart ensures zero downtime — old version continues serving until the new (rolled-back) version passes health checks on each node
- [ ] Version `2.0.0` registered with breaking input resolution change: `openeye serve` detects the input shape mismatch with the current preprocessing pipeline and warns before applying the rollback
- [ ] Semver range `~2.1` with no `2.1.x` versions available: fails with a clear error listing available versions matching `2.x`
- [ ] `openeye model promote` to `production` without going through `staging`: blocked by default. `--skip-staging` flag overrides with a warning in the audit log
- [ ] Promote a model that has no benchmark results (story 187): logs a warning "No benchmark results found for warehouse-safety:2.1.0 — consider running `openeye benchmark` before promoting to production"
- [ ] Clock skew between registry server and serving nodes: audit log timestamps use server-side UTC — client-side timestamps are recorded separately for debugging but not used for ordering
- [ ] Rollback during active A/B test (story 183): no handling for in-progress A/B test when rollback triggers on primary model — test termination behavior undefined
- [ ] Automatic rollback triggered by transient infrastructure issue: error rate spike from downstream dependency (not model fault) could trigger false-positive rollback
- [ ] Audit log storage exhaustion: append-only audit log without rotation or archival could grow unbounded

### Technical Notes

- Versioning metadata is stored in the registry database alongside the model manifest (story 181)
- Rollback triggers a redeployment via the same mechanism as `openeye serve --model` reload — the serving layer watches for tag changes on the registry
- Audit log entries are append-only and include: `actor`, `action`, `version_from`, `version_to`, `timestamp`, `reason`, `metadata`
- Automatic rollback uses Prometheus metrics from the serving layer (story 146) to compute error rate
- Semver resolution uses the `semver` Python library for range matching

## 183. A/B Testing (Model Comparison)

**As an enterprise ML team, I can run shadow/canary model comparisons in production to evaluate new models before full deployment.**

### Acceptance Criteria

- [ ] `openeye serve --model registry://warehouse-safety:2.0.0 --canary registry://warehouse-safety:2.1.0 --canary-weight 10` routes 10% of traffic to the canary model and 90% to the primary
- [ ] `openeye serve --model registry://warehouse-safety:2.0.0 --shadow registry://warehouse-safety:2.1.0` runs the shadow model on 100% of traffic but only returns the primary model's results to the client — shadow results are logged for offline comparison
- [ ] Canary weight is adjustable at runtime via `PUT /api/v1/ab-test/config` without restarting the server: `{"canary_weight": 25}`
- [ ] A/B test dashboard at `GET /api/v1/ab-test/results` returns real-time comparison metrics: latency (p50, p95, p99), throughput, error rate, detection count delta, and confidence distribution per model
- [ ] `openeye ab compare --test-id <id>` generates a CLI report summarizing the A/B test with statistical significance (chi-squared test for detection rate, Mann-Whitney U for latency)
- [ ] Each response in canary mode includes an `X-OpenEye-Model-Version` header indicating which model served the request
- [ ] Shadow mode logs both primary and shadow results to a comparison log file: `--shadow-log shadow_results.jsonl` — each entry has `request_id`, `primary_result`, `shadow_result`, `latency_primary_ms`, `latency_shadow_ms`
- [ ] Automatic canary promotion: `--canary-auto-promote --promote-after 1h --promote-if "error_rate < 0.01 AND p95_latency < 200ms"` promotes the canary to primary after the specified conditions are met for the duration
- [ ] Automatic canary rollback: `--canary-auto-rollback --rollback-if "error_rate > 0.05 OR p95_latency > 500ms"` terminates the canary and reverts to 100% primary
- [ ] Multi-model A/B: supports up to 3 variants simultaneously: `--variant-b registry://model:v2 --variant-b-weight 15 --variant-c registry://model:v3 --variant-c-weight 5`
- [ ] Detection-level comparison: shadow mode aligns detections between primary and shadow models by IoU overlap (threshold: 0.5) and reports added/removed/changed detections per frame
- [ ] A/B test results are exportable: `openeye ab export --test-id <id> --format csv` for offline analysis in notebooks or dashboards

### Edge Cases

- [ ] Canary model crashes during inference: the request is automatically retried on the primary model with no user-visible error. The canary error is logged and the canary error rate metric is incremented
- [ ] Shadow model is significantly slower than primary (>3x latency): shadow inference is cancelled after a configurable timeout (`--shadow-timeout`, default: 5s) and the result is recorded as `timeout` in the shadow log — does not block the primary response
- [ ] Canary weight set to 0: effectively disables the canary without unloading the model — useful for pausing a test while keeping the model warm in memory
- [ ] GPU memory exhaustion from loading two models: if total VRAM is insufficient, the server refuses to start the A/B test and logs the per-model VRAM requirement vs available VRAM. Suggests using shadow mode with sequential inference (`--shadow-sequential`) as a lower-memory alternative
- [ ] Canary model returns a different schema version (e.g., new detection fields): the response is normalized to the primary's schema before returning to the client — new fields are included only in the comparison log
- [ ] Statistical significance: `openeye ab compare` warns if the sample size is too small for meaningful comparison (< 1000 requests) and reports the minimum sample size needed for 95% confidence
- [ ] Network partition between serving node and registry during auto-promote: promotion is deferred until registry connectivity is restored. The canary continues serving at its current weight
- [ ] A/B test with streaming (`/ws` or gRPC): canary routing is decided per-connection, not per-frame — a client connected to the canary stays on the canary for the entire session
- [ ] Shadow log file grows unbounded: `--shadow-log-max-size 1GB --shadow-log-rotate 5` implements log rotation with configurable max size and retained file count
- [ ] Overlapping A/B tests: only one A/B test can be active per serving endpoint at a time. Attempting to start a second test returns `409 Conflict` with the active test ID
- [ ] Latency mismatch biasing results: significantly slower canary model causes different user behavior (timeouts, retries) — skews non-latency metrics
- [ ] Canary auto-promote with insufficient statistical power: time-based gate (`--promote-after 1h`) but no minimum sample size gate — low-traffic endpoints could promote with only 50 requests
- [ ] A/B test across models with different class sets: canary detects additional classes — detection count delta misleading without class-aware comparison

### Technical Notes

- A/B test routing is implemented in the serving layer's request handler, before model inference
- Canary routing uses consistent hashing on `request_id` (or `client_id` if available) for deterministic routing — the same client always hits the same model variant during a test
- Shadow mode runs inference asynchronously in a background thread/task — the primary response is returned immediately
- Comparison metrics are stored in-memory with periodic flush to disk or Prometheus (story 146)
- Detection alignment uses IoU (Intersection over Union) matching from the `torchvision.ops.box_iou` function
- A/B test configuration and results are persisted in the registry database (story 181)

## 184. Custom Model Training Pipeline

**As an enterprise ML team, I can train custom detection models on our proprietary datasets using managed training pipelines.**

### Acceptance Criteria

- [ ] `openeye train --dataset s3://acme/datasets/warehouse-v3 --base-model yolov8n --epochs 100 --output registry://acme/warehouse-safety:1.0.0` launches a training job and registers the result in the model registry (story 181)
- [ ] Training datasets support COCO, YOLO (Ultralytics), Pascal VOC, and Label Studio JSON formats — auto-detected from directory structure or specified via `--format coco`
- [ ] `openeye train --config train.yaml` accepts a full training configuration file specifying model, dataset, hyperparameters, augmentations, and output destination
- [ ] Training supports distributed multi-GPU: `openeye train --gpus 4 --strategy ddp` uses PyTorch DistributedDataParallel across local GPUs
- [ ] Remote training: `openeye train --remote --instance-type gpu.a100.4x` submits the job to a managed cloud training cluster and streams logs back to the CLI
- [ ] `openeye train status <job-id>` shows real-time training progress: epoch, loss, mAP, learning rate, ETA, and GPU utilization
- [ ] `openeye train logs <job-id>` streams training logs (stdout/stderr) for remote jobs
- [ ] `openeye train cancel <job-id>` cancels a running training job and cleans up resources
- [ ] Training checkpoints are saved every N epochs (`--checkpoint-interval 10`) to `--checkpoint-dir` or the registry's artifact storage
- [ ] Best model (highest validation mAP) is automatically selected and registered at the end of training with the `--output` registry path
- [ ] Training supports early stopping: `--early-stopping --patience 15 --min-delta 0.001` stops training when validation mAP plateaus
- [ ] Post-training export: the trained model is automatically exported to ONNX and optionally TensorRT (`--export onnx,tensorrt`) and all formats are registered as variants in the registry
- [ ] Training metrics (loss, mAP, precision, recall per epoch) are logged to experiment tracking (story 189) if configured

### Edge Cases

- [ ] Dataset validation errors (missing images, corrupt annotations, class label mismatches): `openeye train` runs a pre-flight dataset validation check before starting training. Lists all errors with file paths and skips corrupt entries if `--skip-invalid` is set (default: fail on first error)
- [ ] GPU OOM during training: catches `CUDA OutOfMemoryError`, logs the current batch size and GPU memory usage, and suggests reducing `--batch-size` or enabling gradient checkpointing (`--gradient-checkpointing`)
- [ ] Training job interrupted (SIGTERM, spot instance preemption): saves an emergency checkpoint and logs the resume command: `openeye train --resume <job-id>` which loads the latest checkpoint and continues from that epoch
- [ ] Remote training cluster has no available GPUs: the job enters a `QUEUED` state with an ETA based on current queue depth. `openeye train status` shows position in queue
- [ ] Dataset with extreme class imbalance (>100:1 ratio): logs a warning during validation suggesting class weighting (`--class-weights auto`) or oversampling (`--oversample-minority`)
- [ ] Training on a dataset with zero validation split: automatically holds out 20% for validation with a warning, unless `--val-split 0` is explicitly set (in which case, validation metrics are unavailable)
- [ ] Network failure during remote training: the job continues on the remote cluster. `openeye train logs <job-id>` reconnects and resumes log streaming from the last received line
- [ ] ONNX export fails for a custom architecture (unsupported ops): logs the specific unsupported operations and registers only the PyTorch model — does not block the entire pipeline
- [ ] Training with `--base-model` that requires a license agreement (e.g., certain foundation models): prompts the user to accept the license before downloading weights. `--accept-license` flag for non-interactive mode
- [ ] Checkpoint directory runs out of disk space: catches `OSError`, logs the available disk space, and suggests reducing `--checkpoint-interval` or cleaning old checkpoints (`openeye train clean-checkpoints <job-id> --keep-best 3`)
- [ ] NaN/Inf loss during training: no detection when training produces NaN or Inf values — should halt or reduce learning rate
- [ ] Distributed training node failure (DDP): multi-GPU training where one node fails mid-epoch — no recovery for NCCL timeout or partial failure
- [ ] Checkpoint corruption: saved checkpoint corrupted by power loss during write — no integrity validation on save or resume
- [ ] Data pipeline stall: training data loader hangs due to I/O bottleneck or S3 throttling — no timeout or detection for stalled loading

### Technical Notes

- Training orchestration lives in `cli/openeye_ai/training/pipeline.py`
- Local training uses Ultralytics YOLO training API under the hood for YOLO models; custom training loops for other architectures
- Remote training uses a job submission API at `POST /api/v1/training/jobs` on the OpenEye platform backend
- Dataset loaders live in `cli/openeye_ai/training/datasets/` with `coco.py`, `yolo.py`, `voc.py`, `label_studio.py`
- ONNX export uses `torch.onnx.export` with opset 17; TensorRT conversion uses `trtexec` or the TensorRT Python API
- Training configs reference the Ultralytics `cfg/` defaults but allow full override

### Example Config

```yaml
# train.yaml
base_model: yolov8m
dataset:
  path: s3://acme/datasets/warehouse-v3
  format: coco
  train: train/
  val: val/
  test: test/

hyperparameters:
  epochs: 200
  batch_size: 32
  learning_rate: 0.001
  optimizer: AdamW
  weight_decay: 0.0005
  warmup_epochs: 5
  lr_scheduler: cosine

augmentation:
  mosaic: 1.0
  mixup: 0.15
  hsv_h: 0.015
  hsv_s: 0.7
  hsv_v: 0.4
  flipud: 0.5
  fliplr: 0.5

output:
  registry: registry://acme/warehouse-safety:1.0.0
  export: [onnx, tensorrt]
  checkpoint_interval: 10
  early_stopping:
    patience: 20
    min_delta: 0.001

resources:
  gpus: 2
  strategy: ddp
```

## 185. Transfer Learning & Fine-Tuning

**As an enterprise ML team, I can fine-tune pre-trained OpenEye models on domain-specific data (e.g., warehouse safety, retail analytics) to achieve higher accuracy on our use cases.**

### Acceptance Criteria

- [ ] `openeye finetune --base-model registry://openeye/yolov8m-coco:latest --dataset s3://acme/warehouse-data --epochs 50 --output registry://acme/warehouse-yolov8:1.0.0` fine-tunes a pre-trained model on custom data
- [ ] Fine-tuning supports three strategies configurable via `--strategy`: `full` (update all layers), `head-only` (freeze backbone, train detection head), `progressive` (unfreeze layers gradually over epochs)
- [ ] `--strategy head-only` is the default for small datasets (<1000 images), `full` for large datasets (>10,000 images) — heuristic is overridable
- [ ] Learning rate is automatically scaled based on the strategy: head-only uses 10x the backbone LR, progressive uses layer-wise LR decay (`--lr-decay-factor 0.9`)
- [ ] `openeye finetune --base-model registry://openeye/yolov8m-coco:latest --add-classes "forklift,pallet,hard_hat,safety_vest"` extends the model with new detection classes while retaining all original COCO classes
- [ ] `--remove-classes "toilet,teddy bear,hair dryer"` prunes irrelevant COCO classes from the detection head, reducing model size and inference time
- [ ] Fine-tuning preserves the base model's knowledge: on held-out COCO validation images, the fine-tuned model's mAP on retained classes drops by no more than 2% (reported in training summary)
- [ ] Domain-specific augmentation presets: `--augmentation-preset warehouse` applies augmentations suited for indoor warehouse environments (varying lighting, motion blur, occlusion by shelving)
- [ ] `openeye finetune --base-model registry://openeye/grounding-dino:latest --dataset ... --vocab "custom safety equipment"` fine-tunes open-vocabulary models with custom vocabulary
- [ ] Fine-tuning logs are integrated with experiment tracking (story 189) — base model lineage is recorded as a parent experiment
- [ ] The resulting model's metadata in the registry (story 181) includes: base model reference, fine-tuning strategy, dataset reference, class mapping, and original vs fine-tuned mAP comparison
- [ ] `openeye finetune evaluate --model registry://acme/warehouse-yolov8:1.0.0 --dataset s3://acme/warehouse-test` runs evaluation on a test set and reports per-class mAP, precision, recall, and confusion matrix

### Edge Cases

- [ ] Fine-tuning with a very small dataset (<50 images): logs a strong warning about overfitting risk and automatically enables aggressive augmentation, dropout increase, and early stopping with patience 5. Suggests using `--strategy head-only` if not already set
- [ ] Class name collision: if `--add-classes` includes a class name that already exists in the base model (e.g., `person`), the CLI warns that the existing class will be retrained with the new data — not duplicated
- [ ] Base model is in a format that doesn't support fine-tuning (e.g., TensorRT engine): fails with a clear error listing formats that support fine-tuning (PyTorch `.pt`, SafeTensors) and suggests pulling the base model in a compatible format
- [ ] Fine-tuning on a dataset with classes not in `--add-classes`: logs a warning listing unrecognized classes in the dataset annotations and skips those annotations during training
- [ ] Catastrophic forgetting detection: after fine-tuning, the pipeline automatically evaluates on a held-out sample of the base model's original validation set. If mAP drops by more than the `--max-forgetting-threshold` (default: 5%), logs a warning and suggests reducing epochs or using `--strategy progressive`
- [ ] Progressive unfreezing schedule conflict: if `--epochs 10` but the backbone has 50 layers to progressively unfreeze, each layer group unfreezes for fewer than 1 epoch — warns that progressive strategy requires more epochs for this architecture
- [ ] Mixed-precision fine-tuning on a GPU that doesn't support FP16 (e.g., older GTX series): detects the GPU capability and falls back to FP32 with a warning about slower training
- [ ] `--augmentation-preset` combined with manual augmentation overrides: preset values are applied first, then manual overrides replace specific values — logs the final augmentation config for transparency
- [ ] Fine-tuning an open-vocabulary model (grounding-dino) with a dataset that has no text prompts: generates default text prompts from class labels ("a photo of a {class_name}") and logs the generated prompts
- [ ] Resume fine-tuning from a checkpoint of a previous fine-tuning run: `openeye finetune --resume <job-id>` correctly loads the partially fine-tuned weights, optimizer state, and learning rate schedule
- [ ] Base model with backdoor/poisoned weights: fine-tuning a poisoned base model propagates backdoor — no integrity verification beyond registry checksum
- [ ] GPU OOM during fine-tuning: loading large base model + optimizer state for `full` strategy not covered — fine-tuning-specific memory issues
- [ ] Concurrent fine-tuning jobs registering to same output: race condition on registry registration of fine-tuned result

### Technical Notes

- Fine-tuning reuses the training pipeline (story 184) with additional flags for layer freezing and base model lineage
- Layer freezing is implemented via `param.requires_grad = False` on backbone layers, with progressive unfreezing controlled by a callback
- Class addition modifies the detection head's final layer dimensions — weights for existing classes are transferred, new class weights are initialized with Xavier uniform
- Augmentation presets are defined in `cli/openeye_ai/training/augmentation_presets.yaml`
- Knowledge retention evaluation uses a cached subset of the base model's original validation set stored alongside the base model in the registry
- Fine-tuned model lineage is stored as a DAG in the registry: `base_model → fine-tuned_v1 → fine-tuned_v2`

## 186. Active Learning Loop

**As a platform user, the system suggests the most informative frames for human labeling to maximize model improvement with minimal annotation effort.**

### Acceptance Criteria

- [ ] `openeye active-learn --model registry://acme/warehouse-safety:1.0.0 --source rtsp://camera1 --output s3://acme/active-learning/batch-001` monitors a live stream and captures the most informative frames for labeling
- [ ] Informativeness scoring uses multiple strategies configurable via `--strategy`: `uncertainty` (highest model uncertainty), `diversity` (most visually diverse), `hybrid` (combination), `committee` (disagreement between model ensemble)
- [ ] Uncertainty scoring methods: `--uncertainty-method entropy` (prediction entropy), `softmax-margin` (margin between top-2 class probabilities), `mc-dropout` (Monte Carlo dropout variance, N=10 forward passes)
- [ ] `--budget 500` limits the batch to 500 frames per active learning cycle — the most informative frames are selected from all captured candidates
- [ ] `openeye active-learn export --batch batch-001 --format label-studio --output tasks.json` exports the selected frames as pre-annotation tasks for Label Studio (references story 89 export format)
- [ ] `openeye active-learn import --batch batch-001 --annotations labeled.json` imports human-labeled annotations back into the dataset for the next training cycle
- [ ] `openeye active-learn suggest --dataset s3://acme/warehouse-data --model registry://acme/warehouse-safety:1.0.0` retrospectively selects the most informative images from an existing dataset (not live stream)
- [ ] The active learning loop tracks cycle history: `openeye active-learn history` shows each cycle's batch size, labeling progress, model version trained, and mAP improvement
- [ ] Automated full loop: `openeye active-learn auto --source rtsp://camera1 --label-tool label-studio --retrain-on-complete --cycles 5` runs 5 active learning cycles, waiting for human labels between each
- [ ] Selected frames include a `difficulty_score` (0.0–1.0) and `selection_reason` (e.g., "low confidence on class 'forklift'", "novel background not seen in training set") for annotator context
- [ ] Redundancy filter: frames that are visually near-identical (SSIM > 0.95) to already-selected frames are deduplicated — only the most informative variant is kept

### Edge Cases

- [ ] Model returns uniform confidence across all classes (complete uncertainty): these frames are ranked highest by entropy but capped at 20% of the budget to avoid over-representing out-of-distribution garbage frames — diversity scoring balances the selection
- [ ] No frames above the minimum informativeness threshold (`--min-uncertainty 0.3`): the batch is undersized and a message is logged: "Only 127/500 frames met the minimum uncertainty threshold — model may already perform well on this data distribution"
- [ ] `committee` strategy with only one model: falls back to `uncertainty` with a warning that committee scoring requires at least 2 models (`--models model1,model2`)
- [ ] MC-dropout on a model without dropout layers: detects the architecture, logs a warning, and falls back to standard entropy scoring
- [ ] Live stream source goes offline during active learning capture: pauses frame selection, reconnects with exponential backoff, and resumes from the current budget count — does not restart from zero
- [ ] Imported annotations have mismatched class labels compared to the model's class list: logs warnings for each unrecognized class with suggestions (fuzzy match) and skips those annotations unless `--map-classes old_name:new_name` is provided
- [ ] Active learning on a dataset with pre-existing labels: only unlabeled images are considered for selection. If the dataset has partial labels, frames with missing class annotations are scored as uncertain for those classes specifically
- [ ] Frame diversity calculation on large batches (>10,000 candidates): uses feature embedding (ResNet/CLIP) with approximate nearest neighbor (Annoy/FAISS) instead of pairwise SSIM — logs which method is being used
- [ ] Human annotator relabels a frame previously used in training with a different label: the conflict is detected during import and flagged for review rather than silently overwriting the training data
- [ ] Active learning cycle with zero mAP improvement: the system suggests investigating data quality, increasing budget, or switching selection strategy — logged as actionable recommendations in the cycle history
- [ ] Poisoning attacks through active learning: adversary controlling labeling interface intentionally mislabels highest-uncertainty frames — maximally degrades model quality with no annotation anomaly detection
- [ ] Active learning feedback loop bias: repeatedly selecting high-uncertainty frames biases training toward edge cases — no balance between informativeness and representativeness
- [ ] Privacy concerns in selected frames: selected frames may contain sensitive content (faces, license plates) — no PII filtering before export to labeling tools

### Technical Notes

- Active learning engine lives in `cli/openeye_ai/active_learning/engine.py`
- Uncertainty scoring uses the model's raw logits (pre-softmax) for entropy and margin calculations
- MC-dropout temporarily enables dropout layers at inference time via `model.train()` on dropout modules only
- Diversity scoring uses CLIP embeddings from `sentence-transformers` with k-medoids clustering to select diverse representatives
- Frame deduplication uses `skimage.metrics.structural_similarity` or perceptual hashing (dHash) for speed
- The active learning loop integrates with the training pipeline (story 184) for automated retraining
- Label Studio integration uses the Label Studio API for task creation and annotation import

### Technical Notes

- Optimization engine lives in `cli/openeye_ai/optimization/`
- Quantization uses PyTorch's native quantization (`torch.quantization`) for PTQ and QAT; TensorRT's calibrator for TensorRT-specific INT8
- Pruning uses `torch.nn.utils.prune` with custom criteria implementations
- Knowledge distillation uses a custom training loop with KL-divergence loss for logit matching and MSE loss for feature alignment
- CoreML export uses `coremltools`; TFLite export uses `tf.lite.TFLiteConverter`; OpenVINO uses `openvino.tools.mo`
- Optimization profiles are defined in `cli/openeye_ai/optimization/profiles.yaml`
- The optimization pipeline integrates with the model registry (story 181) to register optimized variants with lineage tracking back to the source model
- OTA deployment of optimized models to edge devices references story 134 for update delivery

## 192. Federated Learning

**As an enterprise team, I can train models across distributed edge devices without centralizing sensitive video data, preserving privacy while improving model accuracy.**

### Acceptance Criteria

- [ ] `openeye federated init --name warehouse-fl --base-model registry://acme/warehouse-safety:1.0.0 --rounds 50 --min-clients 5` initializes a federated learning job on the central server
- [ ] `openeye federated join --server https://fl.corp.com --client-id edge-node-01 --data-dir /data/local-annotations/` enrolls an edge device as a federated learning participant
- [ ] The central server orchestrates training rounds: broadcasts the global model → clients train locally on private data → clients send only gradient updates (not raw data) → server aggregates updates → repeat
- [ ] Aggregation strategies are configurable via `--aggregation`: `fedavg` (Federated Averaging), `fedprox` (FedProx with proximal term), `fedyogi` (adaptive optimization), `scaffold` (variance reduction)
- [ ] Differential privacy is enforced on gradient updates: `--dp-epsilon 8.0 --dp-delta 1e-5 --dp-clip-norm 1.0` clips and adds calibrated Gaussian noise to gradients before upload
- [ ] Secure aggregation: `--secure-agg` ensures the server only sees the aggregated gradient, not individual client updates — uses additive secret sharing among clients
- [ ] `openeye federated status <job-name>` shows: current round, participating clients, per-round global mAP, convergence metrics, communication cost (MB sent/received per client)
- [ ] `openeye federated results <job-name>` displays final model metrics, per-client contribution weight, and rounds-to-convergence
- [ ] Client selection: `--client-selection random` (default), `power-of-choice` (select clients with most data), or `stratified` (select clients to balance class distribution across the round)
- [ ] Communication efficiency: gradient compression via `--compress top-k` (send only top-k% of gradient values) or `--compress quantize` (1-bit SGD) to reduce bandwidth by 10–100x
- [ ] The final aggregated model is automatically registered in the model registry (story 181) with federated training metadata: number of rounds, participating clients (anonymized), aggregation strategy, and privacy budget spent
- [ ] `openeye federated simulate --config fl-config.yaml --dataset s3://acme/fl-test-data --partitions 10 --iid false` simulates federated learning locally by partitioning a dataset into non-IID shards across virtual clients
- [ ] Edge devices run local training using the training pipeline (story 184) with device-appropriate batch sizes and precision (story 90 for Jetson optimization)

### Edge Cases

- [ ] Client drops out mid-round (network failure, device reboot): the round proceeds if at least `--min-clients` have submitted updates. The dropped client's partial work is discarded. The client automatically rejoins in the next round and re-downloads the latest global model
- [ ] Client with severely non-IID data (e.g., only "forklift" images): FedAvg may diverge. `--aggregation fedprox --mu 0.1` adds a proximal regularization term to keep client updates close to the global model. The system logs per-client gradient divergence to detect problematic data distributions
- [ ] Malicious or poisoned client sending adversarial gradients: `--robust-agg krum` (Krum) or `--robust-agg trimmed-mean` replaces FedAvg with Byzantine-robust aggregation that tolerates up to `f` malicious clients out of `n` total (configurable via `--max-byzantine`)
- [ ] Differential privacy budget exhausted: after the cumulative privacy budget (epsilon) across all rounds reaches `--dp-max-epsilon` (default: 10.0), training stops with a message: "Privacy budget exhausted after 37 rounds. Final model registered. To continue, increase --dp-max-epsilon (at the cost of weaker privacy guarantees)"
- [ ] Secure aggregation with client dropout: if a client participating in secret sharing drops out before the unmasking phase, the round must be restarted with the remaining clients. After `--max-round-retries` (default: 3), the round is skipped and the global model remains unchanged
- [ ] Gradient compression artifacts: top-k compression with very aggressive k (e.g., 1%) may cause slow convergence. The system tracks rounds-to-convergence and logs a warning if it exceeds 2x the expected rate, suggesting a higher k value
- [ ] Heterogeneous client hardware (mix of Jetson Nano, Orin, x86 workstations): local training epochs are adaptive — `--local-epochs auto` assigns more epochs to faster clients and fewer to slower ones, weighted by compute capacity reported during enrollment
- [ ] Client data changes between rounds (new annotations added): the client includes a data version hash in its round submission. The server tracks data evolution per client for provenance
- [ ] Global model accuracy decreases after a round (negative transfer): the server compares post-aggregation mAP to pre-round mAP on a held-out validation set. If mAP drops by more than `--max-round-regression` (default: 2%), the round's update is rejected and the previous global model is retained
- [ ] Cross-silo vs cross-device: `--mode cross-silo` (few reliable clients with large datasets, e.g., hospital branches) uses synchronous rounds. `--mode cross-device` (many unreliable clients with small datasets, e.g., IoT cameras) uses asynchronous aggregation with staleness-weighted averaging
- [ ] Regulatory compliance: `openeye federated audit <job-name>` generates a compliance report documenting: data never left client devices, privacy mechanisms applied, aggregation strategy, and per-round privacy accounting — suitable for GDPR/HIPAA documentation
- [ ] Training data exfiltration via gradient inversion: gradient updates can be inverted to reconstruct training images — no defense beyond DP noise
- [ ] Client impersonation / Sybil attack: no authentication beyond `--client-id` — adversary enrolls multiple fake clients to dominate aggregation
- [ ] Gradient compression interacting with DP: combining top-k compression with DP noise may invalidate privacy guarantees if compression applied before noise — order of operations unspecified
- [ ] TOCTOU on min-clients threshold: between checking min-clients joined and starting round, clients may disconnect — round begins with fewer than minimum

### Technical Notes

- Federated learning server component lives in `backend/federated/server.py`; client component in `cli/openeye_ai/federated/client.py`
- Communication uses gRPC with TLS for gradient upload/download between clients and server
- Gradient serialization uses Protocol Buffers with optional zlib compression
- Differential privacy implementation uses Opacus (`opacus`) for per-sample gradient clipping and noise calibration
- Secure aggregation protocol follows the Bonawitz et al. (2017) design with Shamir secret sharing
- Byzantine-robust aggregation uses the Multi-Krum algorithm from Blanchard et al. (2017)
- Federated simulation uses PyTorch's `DataLoader` with custom `Sampler` implementations for IID and non-IID partitioning (Dirichlet distribution with configurable alpha for non-IID-ness)
- The federated client on edge devices reuses the training pipeline (story 184) and edge device support (story 90) for local training
- Model registry integration (story 181) stores the federated model with `training_method: federated` metadata and a link to the federated job's audit log
